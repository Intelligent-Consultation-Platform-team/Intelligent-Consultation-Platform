# 智能咨询平台测试任务汇报

## 一、汇报目的

本文档用于总结当前项目已完成的测试工程建设情况，说明各阶段测试任务的目标、覆盖范围、执行结果以及配套的文档与自动化配置，便于后续维护、验收和持续迭代。

## 二、测试工程总体目标

本次测试工程的目标是围绕“后端业务稳定性、前端交互正确性、端到端流程可用性、持续集成可验证性”四个层面，建立一套可运行、可追踪、可扩展的测试体系。

整体上，测试任务按照 Phase 0 到 Phase 7 逐步推进，形成了从基础测试准备、后端核心功能测试、前端基础逻辑测试，到 E2E、覆盖率和 CI 收尾的完整链路。

- Phase 0：测试环境与基础设施准备
- Phase 1：后端基础单元测试与公共工具测试
- Phase 2：后端业务服务测试
- Phase 3：后端接口与异常处理测试
- Phase 4：后端 AI 问诊模块单元测试与控制器测试
- Phase 5：前端 Vitest 单元测试
- Phase 6：Playwright 端到端测试
- Phase 7：覆盖率、CI 与测试文档收尾

## 三、各阶段测试任务说明

### 3.1 Phase 0：测试环境与基础设施准备

#### 任务目标

为后续测试建设准备基础环境、依赖和测试策略，确保后续阶段能够稳定落地。

#### 主要内容

- 梳理项目当前测试结构
- 确认后端使用 JUnit、Mockito、Spring Boot Test 等测试框架
- 确认前端测试将采用 Vitest 和 Playwright
- 确认测试数据、测试配置和环境变量的约定方式
- 准备测试计划文档和分阶段执行策略

#### 阶段意义

Phase 0 的作用是把测试工程的基础铺好，避免后续阶段出现“没有统一入口、没有统一策略、没有统一环境”的问题。

---

### 3.2 Phase 1：后端基础单元测试与公共工具测试

#### 任务目标

先验证后端底层公共能力是否稳定，包括工具类、异常处理和拦截器等基础设施。

#### 已完成内容

- `JwtUtilsTest`
  - 验证 JWT 生成、解析和有效期逻辑

- `PasswordUtilsTest`
  - 验证密码加密和校验逻辑

- `TokenInterceptorTest`
  - 验证登录态拦截和 Token 识别逻辑

- `GlobalExceptionHandlerTest`
  - 验证业务异常与系统异常的统一响应格式

#### 阶段意义

Phase 1 确保后端公共底座可用，是后续业务测试的前提。

---

### 3.3 Phase 2：后端业务服务测试

#### 任务目标

对核心业务服务层进行单元测试，验证主要业务规则、数据流转和状态变化。

#### 已完成内容

- `UsersServiceImplTest`
  - 验证用户注册、登录、重置密码、逻辑删除和参数校验

- `AppointmentsServiceImplTest`
  - 验证预约挂号相关业务流程

- `PatientAccountServiceImplTest`
  - 验证患者账户、充值、支付和余额查询逻辑

#### 阶段意义

Phase 2 负责覆盖系统最核心的业务规则，确保服务层行为稳定、可回归、可维护。

---

### 3.4 Phase 3：后端接口与启动测试

#### 任务目标

验证后端 Web 层接口的请求/响应表现，以及 Spring Boot 基础启动能力。

#### 已完成内容

- `AuthControllerTest`
  - 验证注册、登录接口的请求/响应格式
  - 验证业务异常的响应映射

- `IntelligentConsultstionPlatformApplicationTests`
  - 验证 Spring Boot 上下文是否能正常启动

#### 阶段意义

Phase 3 将服务层能力连接到接口层，并通过冒烟测试确认整体应用基础设施可正常加载。

---

### 3.5 Phase 4：后端 AI 问诊测试

#### 任务目标

为 AI 问诊模块补充高信号、确定性的测试，重点覆盖服务逻辑、DeepSeek 调用兜底行为和控制器接口映射。

- `AiConsultationServiceTest`

  - 验证发送消息并落库的流程
  - 验证无权限访问会话时的拒绝行为
  - 验证会话详情映射
  - 验证消息分页
  - 验证关闭会话
  - 验证我的会话查询
  - 验证管理员会话查询
  - 验证风险查询默认值

- `AiConsultationControllerTest`
  - 验证创建会话接口
  - 验证发送消息接口
  - 验证会话详情接口
  - 验证消息列表接口
  - 验证关闭会话接口
  - 验证我的会话接口
  - 验证管理员查看全部会话接口
  - 验证风险查询接口
  - 验证上传接口兜底提示

#### 执行结果

- 已完成并通过验证
- 后端 AI 问诊相关测试可稳定执行，不依赖外部 DeepSeek 服务
- 相关测试代码已补充中文注释，便于后续维护

## 六、当前测试工程总结

目前项目测试体系已经从“局部补测”完成了向“体系化测试工程”的转变，主要体现为：

- 有明确的阶段划分
- 有后端、前端、E2E 三层测试
- 有覆盖率产出
- 有 CI 自动执行
- 有测试运行文档
- 关键测试代码已补充中文注释，便于团队协作与后续维护

整体来看，当前测试工程已经能够支持项目在日常开发、回归验证和持续集成场景下的稳定使用。

## 七、后续建议

- 后续新增业务功能时，遵循“先补测试，再改实现”的方式
- 新增前端页面后，优先补充 Vitest 与 Playwright 的对应测试
- 新增后端接口时，优先补充 Service 与 Controller 两层测试
- 持续关注 E2E 场景的稳定性，避免过度依赖页面样式或固定文案
- 定期查看 JaCoCo 覆盖率，确保核心业务逻辑不会出现明显回退
